## 1. 首发结论

第一版继续采用微信小程序 `web-view` 承载现有 Humi H5，不进行原生页面重写。

首发只验证一件事：

> 用户是否愿意反复使用 Humi 解决“今晚吃什么”。

首发核心链路：

```text
打开 Humi
→ 【今晚】看到推荐
→ 今晚就做 / 换一组 / 不想吃
→ 问问大家想吃啥
→ 勾选要做的菜
→ 【清单】自动生成食材并可分享认领
→ 【我的家】沉淀饭线索
```

域名备案完成前，不阻塞 H5 产品验收、小程序壳检查、审核材料准备和埋点验证。

## 2. MVP 功能边界

### P0：首发必须可用

| 能力 | 首发要求 | 当前状态 |
| --- | --- | --- |
| 小程序启动 | 打开后直接进入 Humi【今晚】 | 已具备壳 |
| 游客体验 | 不登录即可获得推荐 | 已具备 |
| 今晚推荐 | 可生成、换一组、反馈、接受 | 已具备 |
| 本地兜底 | 云端推荐失败仍能换到另一组 | 已具备 |
| 今日菜单与三餐计划 | 接受推荐后可查看和调整份数；早餐轻记录、午餐记来源、晚饭完整闭环 | 已具备 |
| 食材清单 | 食材、调料分区，可折叠和勾选 | 已具备 |
| 想连排几天 | 作为【今晚】辅助入口，可选排几天并汇总清单 | 已具备 |
| 自己挑 | 推荐不合适时可自行选择 | 已具备 |
| 家庭协作 | 感觉征集、邀请家人、买菜认领和想吃池子可走小程序卡片 | 已具备 |
| 本地保存 | 游客刷新后核心数据仍保留 | 待微信真机验证 |
| 来源埋点 | 记录 `wechat-miniprogram` 渠道 | 已具备 |
| 错误降级 | 断网或接口失败时有生活化提示 | 待微信真机验证 |

### P0.5：首发灰度优先能力

| 能力 | 策略 | 原因 |
| --- | --- | --- |
| 微信登录 | 小程序主身份方案，使用 `wx.login` 通过服务端换取微信身份并映射 Humi 用户 | 符合国内小程序习惯 |
| 手机号绑定 | 用户需要换设备找回、家庭协作、后续支付或会员时再引导绑定 | 比邮箱更符合国内账号找回习惯 |
| 登录后置 | 只有保存、同步、家庭协作或跨设备恢复时才要求登录 | 先让用户获得“晚饭有了”的价值 |

P0.5 不影响游客核心链路，但会影响正式灰度留存和家庭协作质量。微信登录已由 Humi API 承接；游客模式保留为用户跳过登录或登录失败时的兜底。

### P1：可进入首发，但不能阻塞首发

| 能力 | 首发策略 | 风险 |
| --- | --- | --- |
| 邮箱登录 | 不进入普通用户界面；仅通过隐藏开发参数保留测试能力，海外市场再重新设计 | 国内用户习惯弱，WebView 邮件回跳不稳定 |
| 云端同步 | 已登录用户使用；游客不依赖它完成核心链路 | Supabase 网络与 session |
| 家庭画像 | 登录用户可完善；游客推荐不强制填写 | 登录依赖 |
| 后台已有 | 不做独立库存页；由清单勾选、做饭确认和轻确认隐形维护 | 已具备 |
| 营养与月历 | 营养页按三餐计划做估算参考；仍不作为医疗或专业营养建议 | 非核心价值 |
| 海报预览 | 生成必须可用 | Canvas 和图片加载 |
| 保存与分享海报 | 真机可用则保留；否则提供长按图片或文案复制提示 | WebView 下载、文件分享能力 |

### P2：首发明确不做

- 原生【今晚】、原生菜谱页或原生食材清单。
- 微信订阅消息。
- 微信支付、会员、商城。
- 原生家庭邀请和菜单导入。
- App Store 或 Capacitor 打包。
- 社区、UGC 和公开内容广场。

## 3. 当前技术结构

```text
微信小程序
└── miniprogram/ WebView 壳
    └── 正式 HTTPS H5
        ├── React + Vite
        ├── 游客 localStorage
        ├── Humi API 微信登录与家庭状态同步
        ├── Humi API 家庭协作接口
        └── DeepSeek 精准推荐（服务端额度闸门，失败时本地规则兜底）
```

现有小程序壳：

```text
miniprogram/
  app.json
  project.config.json
  utils/config.js
  pages/index/
```

H5 通过查询参数识别来源：

```text
?channel=wechat-miniprogram
```

小程序 WebView 内会跳过 PWA service worker 注册，避免缓存版本与小程序刷新互相干扰。

## 4. WebView 兼容风险

### 高风险

#### 4.1 身份登录

- 小程序环境不再把邮箱作为主登录方式。
- 邮箱入口不服务普通用户；仅通过隐藏开发参数服务后台测试和开发验证。
- 微信登录已通过 Humi API 换取微信身份，并与 Humi 用户、家庭空间建立映射。
- 手机号绑定由用户主动授权，并通过服务端完成手机号换取与绑定。
- 发布决策：游客核心链路不依赖登录；家庭协作、状态同步和正式成员身份依赖微信登录。
- 降级策略：微信登录服务短暂失败时提示重试；手机号绑定失败或用户拒绝授权时回到 Humi，不影响核心功能。

#### 4.2 海报保存与分享

- 当前海报使用浏览器 Canvas 生成 PNG。
- 分享依赖 `navigator.share`，保存依赖浏览器下载。
- 微信 WebView 对文件分享和下载支持可能与 Safari 不一致。
- 发布决策：预览生成是 P0；系统分享和 PNG 下载是 P1。
- 降级策略：显示海报预览，提示用户长按保存；必要时退回复制文字。

### 中风险

#### 4.3 本地数据持久化

- 游客数据依赖 localStorage。
- 必须验证退出小程序、微信回收 WebView、重新打开后的数据保留情况。
- 若数据被清理，首发文案不能承诺跨设备或长期保存。

#### 4.4 Humi API 请求

- 微信登录、状态同步、家庭邀请、感觉征集、买菜认领和精准推荐会访问 `https://api.humi-home.com`。
- 需要在真机网络下验证 request 合法域名、session 恢复和小程序卡片落地页。
- 精准推荐接口失败时必须立即回退基础推荐；基础推荐和感觉征集入口不得被精准 API 成本或额度阻断。

#### 4.5 外部协议页面

- 隐私政策和用户协议必须使用正式域名并可独立打开。
- 页面中的“返回 Humi”链接必须指向正式 H5 路径。

### 低风险

- CSS safe area 和底部导航。
- 图片与字体加载。
- 页面滚动和抽屉交互。
- PWA 安装提示：小程序渠道应隐藏或不触发。

## 7. 真机验收矩阵

> 当前用法：本节用于审核通过/发布后的真机 P0 验收参考。提审前 P0/P1 已完成；发布后的真机验收结果登记到 `docs/humi-1.1-release-evidence-log.md`。

- [ ] 首次打开直接进入【今晚】，无登录墙。

- [ ] 底部导航不被安全区遮挡。

- [ ] “换一组”不会重复上一组。
- [ ] 云端失败时能切到本地另一组。
- [ ] 点击“就吃这组”后进入今晚菜单。

### 菜单与清单

- [ ] 调整份数不会误触或卡顿。

## 8. 首发数据指标

- 打开用户数。

- 推荐接受率。

## 10. 后续原生化顺序

只有 WebView MVP 获得真实留存后再原生化：

1. 【今晚】推荐。
2. 食材清单。
