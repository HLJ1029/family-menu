# Humi 微信小程序迁移与首发清单 V2

更新日期：2026-06-07

本文档是 Humi 小程序首发的执行基线。其他路线图只保留方向，具体范围、阻塞项和验收结果以本文为准。

## 1. 首发结论

第一版继续采用微信小程序 `web-view` 承载现有 Humi H5，不进行原生页面重写。

首发只验证一件事：

> 用户是否愿意反复使用 Humi 解决“今晚吃什么”。

首发核心链路：

```text
打开 Humi
→ 帮我安排晚饭
→ 换一组并反馈
→ 就吃这组
→ 查看食材清单
→ 生成或分享清单
```

域名备案完成前，不阻塞 H5 产品验收、小程序壳检查、审核材料准备和埋点验证。

## 2. MVP 功能边界

### P0：首发必须可用

| 能力 | 首发要求 | 当前状态 |
| --- | --- | --- |
| 小程序启动 | 打开后直接进入 Humi 首页 | 已具备壳 |
| 游客体验 | 不登录即可获得推荐 | 已具备 |
| 今晚推荐 | 可生成、换一组、反馈、接受 | 已具备 |
| 本地兜底 | 云端推荐失败仍能换到另一组 | 已具备 |
| 今晚菜单 | 接受推荐后可查看和调整份数 | 已具备 |
| 食材清单 | 食材、调料分区，可折叠和勾选 | 已具备 |
| 一周计划 | 可查看并把菜加入本周 | 已具备 |
| 自己挑 | 推荐不合适时可自行选择 | 已具备 |
| 本地保存 | 游客刷新后核心数据仍保留 | 待微信真机验证 |
| 来源埋点 | 记录 `wechat-miniprogram` 渠道 | 已具备 |
| 错误降级 | 断网或接口失败时有生活化提示 | 待微信真机验证 |

### P0.5：小程序正式验证前优先补齐

| 能力 | 策略 | 原因 |
| --- | --- | --- |
| 微信登录 | 小程序主身份方案，使用 `wx.login` 通过服务端换取微信身份并映射 Humi 用户 | 符合国内小程序习惯 |
| 手机号绑定 | 用户需要换设备找回、家庭协作、后续支付或会员时再引导绑定 | 比邮箱更符合国内账号找回习惯 |
| 登录后置 | 只有保存、同步、家庭协作或跨设备恢复时才要求登录 | 先让用户获得“晚饭有了”的价值 |

P0.5 不影响游客核心链路，但会影响正式灰度留存和家庭协作质量。若开发排期紧，仍可先用游客模式提交小范围体验版；正式对外验证前，应优先完成微信登录。

### P1：可进入首发，但不能阻塞首发

| 能力 | 首发策略 | 风险 |
| --- | --- | --- |
| 邮箱登录 | 降级为 Web/PWA 和开发测试入口；小程序内默认不作为主登录 | 国内用户习惯弱，WebView 邮件回跳不稳定 |
| 云端同步 | 已登录用户使用；游客不依赖它完成核心链路 | Supabase 网络与 session |
| 家庭画像 | 登录用户可完善；游客推荐不强制填写 | 登录依赖 |
| 厨房库存 | 可使用，但不作为首发主任务 | 页面较深 |
| 营养与月历 | 保留入口，不作为审核演示主路径 | 非核心价值 |
| 海报预览 | 生成必须可用 | Canvas 和图片加载 |
| 保存与分享海报 | 真机可用则保留；否则提供长按图片或文案复制提示 | WebView 下载、文件分享能力 |

### P2：首发明确不做

- 原生首页、原生菜谱页或原生食材清单。
- 微信订阅消息。
- 微信支付、会员、商城。
- 家庭邀请和分享后自动导入菜单。
- App Store 或 Capacitor 打包。
- 社区、UGC 和公开内容广场。

## 3. 当前技术结构

```text
微信小程序
└── miniprogram/ WebView 壳
    └── 正式 HTTPS H5
        ├── React + Vite
        ├── 游客 localStorage
        ├── Supabase 登录与同步
        ├── Supabase Edge Function
        └── DeepSeek 推荐（失败时本地规则兜底）
```

现有小程序壳：

```text
miniprogram/
  app.json
  project.config.json
  utils/config.js
  pages/index/
```

H5 通过查询参数识别来源：

```text
?channel=wechat-miniprogram
```

小程序 WebView 内会跳过 PWA service worker 注册，避免缓存版本与小程序刷新互相干扰。

## 4. WebView 兼容风险

### 高风险

#### 4.1 身份登录

- 小程序环境不再把邮箱作为主登录方式。
- 邮箱入口只服务 Web/PWA、后台测试和开发验证。
- 微信登录需要新增服务端换取微信身份，并与 Humi 用户、家庭空间建立映射。
- 手机号绑定需要用户主动授权，并通过服务端完成手机号换取与绑定。
- 发布决策：游客核心链路不依赖登录；小程序正式验证前优先补齐微信登录。
- 降级策略：微信登录未完成时，小程序先隐藏登录入口，使用游客本地模式；不要展示不可用的微信或手机号按钮。

#### 4.2 海报保存与分享

- 当前海报使用浏览器 Canvas 生成 PNG。
- 分享依赖 `navigator.share`，保存依赖浏览器下载。
- 微信 WebView 对文件分享和下载支持可能与 Safari 不一致。
- 发布决策：预览生成是 P0；系统分享和 PNG 下载是 P1。
- 降级策略：显示海报预览，提示用户长按保存；必要时退回复制文字。

### 中风险

#### 4.3 本地数据持久化

- 游客数据依赖 localStorage。
- 必须验证退出小程序、微信回收 WebView、重新打开后的数据保留情况。
- 若数据被清理，首发文案不能承诺跨设备或长期保存。

#### 4.4 Supabase 请求

- 登录、同步、推荐和埋点会访问 Supabase。
- 需要在真机网络下验证请求、Edge Function 和 session 恢复。
- 推荐接口失败时必须立即回退本地推荐。

#### 4.5 外部协议页面

- 隐私政策和用户协议必须使用正式域名并可独立打开。
- 页面中的“返回 Humi”链接必须指向正式 H5 路径。

### 低风险

- CSS safe area 和底部导航。
- 图片与字体加载。
- 页面滚动和抽屉交互。
- PWA 安装提示：小程序渠道应隐藏或不触发。

## 7. 真机验收矩阵

每项使用 iPhone 微信至少测试一次；关键链路建议再用一台不同型号手机复测。

### 启动与恢复

- [ ] 首次打开直接进入首页，无登录墙。
- [ ] 冷启动、热启动和返回小程序均正常。
- [ ] 底部导航不被安全区遮挡。
- [ ] 页面刷新后仍能使用。
- [ ] 关闭再打开后，游客菜单和清单按预期保留。

### 核心推荐

- [ ] 点击“帮我安排晚饭”有即时加载反馈。
- [ ] 云端推荐成功时结果可接受。
- [ ] “换一组”不会重复上一组。
- [ ] 云端失败时能切到本地另一组。
- [ ] 点击“就吃这组”后进入今晚菜单。

### 菜单与清单

- [ ] 调整份数不会误触或卡顿。
- [ ] 食材清单默认折叠，页面不过长。
- [ ] 食材和调料分区正确。
- [ ] 同名调料不重复。
- [ ] 勾选、恢复、加入库存正常。

### 分享

- [ ] 今晚菜单海报可以生成。
- [ ] 本周菜单海报可以生成。
- [ ] 购物清单海报可以生成。
- [ ] 图片加载失败时仍有可理解的海报或提示。
- [ ] 长按、分享或保存至少有一种可用路径。

### 登录与云端

- [ ] 游客模式无需登录即可完成核心链路。
- [ ] 小程序不把邮箱作为主登录入口。
- [ ] 微信登录可创建或恢复 Humi 用户。
- [ ] 手机号绑定只在用户主动授权后触发。
- [ ] session 刷新后恢复。
- [ ] 家庭数据不会串到其他账号。
- [ ] 若微信登录未完成，首发隐藏登录入口并保留游客体验。

### 审核表达

- [ ] 页面不出现 DeepSeek、Supabase、PWA、模型、规则引擎或 fallback。
- [ ] 不宣称医疗、诊断或专业营养建议。
- [ ] 隐私政策与实际收集的数据一致。
- [ ] 审核员无需注册即可完成核心体验。

## 8. 首发数据指标

首批灰度只看以下指标：

- 打开用户数。
- 推荐请求次数。
- 换一组率。
- 推荐接受率。
- 接受后查看食材清单的比例。
- 海报预览和分享触发率。
- 次日与 7 日留存。
- 推荐接口错误率和本地兜底率。

第一阶段参考目标：

- 推荐接受率大于 50%。
- 7 日留存大于 25%。
- 核心链路无阻塞错误。
- 至少 10 个家庭完成一次“推荐 → 接受 → 清单”。

- [ ] 无技术化用户文案。

- [ ] 已准备可回滚的上一版 H5。

## 10. 后续原生化顺序

只有 WebView MVP 获得真实留存后再原生化：

1. 首页推荐。
2. 食材清单。
3. 菜谱详情。
4. 手机号绑定、家庭身份和账号找回。
5. 家庭画像与协作。
6. 分享卡片和菜单导入。

原生化的触发条件不是“WebView 看起来不像小程序”，而是某项 WebView 限制已经明显影响留存、转化或审核。
