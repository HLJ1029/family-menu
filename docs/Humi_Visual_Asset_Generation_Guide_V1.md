# Humi Visual Asset Generation Guide V1

## Purpose

This is the single entry point for generating visual assets for Humi.

Before generating or modifying any Humi image, read this file and the referenced design documents.

The workflow is:

1. OpenClaw prepares visual assets and previews.
2. The product owner reviews and confirms the visual direction.
3. Codex validates the files, connects them to product data, tests the product, and publishes the update.

Do not directly replace production assets before visual confirmation.

## Brand Basics

- Brand name: `Humi`
- Slogan: `让每顿饭都有安排`
- UI primary: `#111111`
- Primary text: `#111111`
- UI background: `#FFFFFF`
- Dish image background: warm off-white paper tone is allowed inside food artwork only
- Brand character: black-and-white lifestyle people, warm, modern, restrained, family-oriented, lightly editorial

Deprecated for Humi UI and poster assets:

- Acid green as a primary brand surface.
- Monster, seed, plant-growth, anime, sticker, or childish character systems.
- SVG-like placeholder line art as final production illustration.

Official app icons:

- `public/icons/humi-icon-1024.png`
- `public/icons/humi-icon-512.png`
- `public/icons/humi-icon-192.png`
- `public/icons/humi-apple-touch-icon.png`

Do not recreate or replace the Humi icon unless the product owner explicitly requests a brand redesign.

## Asset Types

### 1. Dish Images

Dish images are reusable product assets.

Production directory:

```text
public/assets/dishes/
```

Thumbnail directory:

```text
public/assets/dishes/thumbs/
```

Recipe data:

```text
data/recipes.json
```

Current generation script:

```text
scripts/generate-dish-images.mjs
```

Thumbnail script:

```text
scripts/generate-dish-thumbnails.mjs
```

#### Dish Image Style

- One Chinese home-style dish.
- One plate or bowl centered in the image.
- 45-degree overhead view.
- Warm off-white paper background.
- Hand-painted watercolor color.
- Fine ink outline.
- Light paper texture.
- Soft restrained shadow.
- Consistent plate scale and camera angle.
- Clear food structure and recognizable ingredients.
- Suitable for square app card cropping.

#### Dish Image Output

- Size: `1024 x 1024`
- Format: PNG
- No watermark
- No text
- No dish name
- No UI
- No labels or buttons
- No chopsticks, hands, people, tablecloth, or decorative props
- No photorealistic restaurant photography
- No 3D-rendered appearance

#### File Naming

Use the recipe id as a lowercase kebab-case filename:

```text
tomato-egg.png
mapo-tofu.png
steamed-sea-bass.png
```

Do not invent a second filename for an existing recipe id.

Do not overwrite an existing production image without producing a comparison preview first.

### 2. Poster Images

Production posters are not generated by an image-generation API.

They are rendered locally from:

1. Humi recipe data
2. Existing Humi dish images
3. Canvas templates
4. Humi brand elements

Read the full poster system before working on poster visuals:

```text
docs/Humi_Poster_Design_System_V1.md
```

OpenClaw may create static poster previews for design discussion, but must not replace the production Canvas templates directly.

Codex is responsible for implementing confirmed poster designs in:

```text
src/lib/posters.js
```

## Preview And Review Rules

For visual or aesthetic work:

1. Generate preview images first.
2. Put previews outside production asset paths, or in a clearly named draft directory.
3. Show the previews to the product owner.
4. Wait for explicit confirmation.
5. Only then prepare final production files.

Recommended temporary preview directory:

```text
/private/tmp/humi-visual-previews/
```

If OpenClaw cannot use that directory, use:

```text
artifacts/visual-previews/
```

Do not commit draft previews unless the product owner asks to preserve them.

## Dish Image Acceptance Checklist

Before handing an image to Codex, verify:

- The dish matches its recipe name.
- Main ingredients are visually recognizable.
- The image belongs to the same family as existing Humi dish images.
- The background is warm and uncluttered.
- The plate or bowl is centered and not cropped awkwardly.
- There is no text, watermark, UI, hand, or person.
- The image remains clear at mobile-card size.
- The filename matches the recipe id.
- A comparison preview was approved if replacing an existing asset.

## Handoff To Codex

When an asset is approved, provide:

1. Recipe id or asset purpose.
2. Final absolute file path.
3. Whether this is a new image or a replacement.
4. The matching recipe name.
5. Any crop or display requirement.
6. The approved preview or approval note.

Codex will then:

1. Validate file dimensions and format.
2. Place or confirm the asset in the production directory.
3. Update recipe data or manifests if needed.
4. Generate thumbnails.
5. Run data validation and build.
6. Test the asset in the product.
7. Commit, push, and verify deployment.
