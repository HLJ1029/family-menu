# Humi 小程序 MVP 准备方案

## 当前策略

先用小程序 `web-view` 承载现有 Humi H5，快速验证“今晚吃什么”和家庭协作这个核心价值。暂不全量重写小程序。登录策略为游客优先、微信登录后置；分享出去的感觉征集、家庭邀请和买菜认领均使用小程序卡片落地到对应 H5 页面。

更细的上线材料、域名、后台配置和审核清单见 `docs/miniprogram-launch-readiness.md`。

## 为什么先 web-view

- 当前 React 版本已经打通【今晚】推荐、菜单、清单、家庭协作和后台已有隐形维护。
- V3 品牌升级后，PWA、小程序、后续 iOS 可以共用同一套产品语言。
- web-view 壳最快，适合在微信生态里做早期 MVP 验证。
- 真正确定用户愿意使用后，再重写小程序核心页。

## 需要准备

- 正式 HTTPS 域名，不建议长期依赖 GitHub Pages 域名。
- 小程序后台配置业务域名。
- 域名校验文件可放到站点根目录。
- 隐私政策和用户协议：
  - `/family-menu/privacy.html`
  - `/family-menu/terms.html`
- 小程序普通启动先进入 Humi【今晚】，不先进入登录页。
- 页面内不要出现旧品牌名、模型名、后端服务名、`PWA`、`规则引擎` 等技术词。

## 小程序第一版范围

- 小程序壳打开 Humi【今晚】。
- 用户能看到“今晚吃什么？”和今晚推荐。
- 用户能查看“自己挑”、“想连排几天”和【清单】。
- 感觉征集、家庭邀请和买菜认领通过小程序卡片进入 H5 落地页，家人轻参与不先撞登录墙。
- 保存、微信登录、家庭协作沿用 H5 + Humi API 现有能力。
- 微信登录作为小程序主身份方案，已由 Humi API 承接。
- 手机号绑定作为账号找回、家庭协作和后续支付/会员的补充身份。
- 邮箱登录只作为 Web/PWA 和开发测试入口，小程序内不作为主登录。

## 第二阶段再考虑

- 原生小程序重写【今晚】。
- 原生小程序重写菜单详情和食材清单。
- 完成手机号绑定的正式灰度验证。
- 接入订阅消息、会员或更深的家庭权限。

## 品牌约束

- 小程序、PWA、后续 iOS 统一叫「Humi」。
- 核心问题统一是「今晚吃什么？」。
- 产品语气保持温暖、家庭感、烟火气、轻陪伴。
- 不对用户展示模型、供应商、接口、规则、fallback 等实现细节。
