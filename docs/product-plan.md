# 家庭点单网站产品企划书

## 1. 项目定位

本项目是一个面向家庭日常做饭场景的“点菜 + 买菜 + 菜谱指导”网站。

用户进入网站后，可以像点外卖一样选择自己想吃的菜。系统会根据所选菜品自动展示制作这道菜需要购买的原材料、调料、预计用量，并提供清晰的做法步骤。后续可以扩展为一周菜单规划、家庭成员口味偏好、购物清单合并、库存管理、营养统计等功能。

项目初期不追求复杂电商闭环，优先做成一个好用的家庭做饭决策工具。

## 2. 背景与问题

很多人在家做饭时会遇到几个常见问题：

1. 不知道今天吃什么。
2. 想吃某道菜，但不知道需要买什么材料。
3. 买菜时容易漏买调料或配菜。
4. 搜菜谱时信息太散，步骤复杂，不适合快速执行。
5. 家庭多人吃饭时，菜品搭配和购物清单难以统一。

本产品希望把“想吃什么”和“需要买什么、怎么做”连起来，让用户从点菜到买菜再到做饭的流程更顺。

## 3. 目标用户

### 主要用户

- 不太会做饭，但愿意在家做饭的新手。
- 家庭主厨，例如父母、年轻夫妻、合租人群。
- 每天需要决定“今天吃什么”的普通家庭用户。

### 次要用户

- 想进行一周食谱规划的人。
- 想控制饮食、记录营养的人。
- 想把家庭常吃菜整理成私有菜谱库的人。

## 4. 核心价值

用户不需要先懂菜谱，也不需要自己拆解采购清单。只要选择想吃的菜，系统就自动给出：

- 需要购买的食材。
- 需要准备的调料。
- 每种材料的建议用量。
- 是否为常见家庭库存，例如盐、酱油、油。
- 做法步骤。
- 难度、耗时、适合人数。
- 可选替代食材。

一句话总结：

让用户从“我想吃西红柿炒鸡蛋”快速得到“我要买什么、怎么做”。

## 5. MVP 最小可行版本

第一版建议只做最核心的闭环，不要一开始做太复杂。

### MVP 功能范围

1. 首页展示菜品列表。
2. 支持搜索菜品名称。
3. 用户点击菜品后进入菜品详情页。
4. 菜品详情页展示：
   - 菜名。
   - 图片。
   - 简介。
   - 适合人数。
   - 预计耗时。
   - 制作难度。
   - 食材清单。
   - 调料清单。
   - 做法步骤。
5. 用户可以把菜品加入“今日菜单”。
6. 系统根据今日菜单合并生成“购物清单”。
7. 购物清单中相同食材自动合并，例如两个菜都需要鸡蛋时合并显示。
8. 支持勾选购物清单中的已购买项目。

### MVP 暂不做

- 在线支付。
- 生鲜配送。
- 用户登录注册。
- AI 自动生成菜谱。
- 复杂营养计算。
- 多端 App。
- 商家后台。

这些功能可以后续再加，第一版先验证用户是否真的愿意用它来决定吃什么和买什么。

## 6. 用户流程

### 核心流程

1. 用户进入首页。
2. 浏览或搜索想吃的菜。
3. 点击“西红柿炒鸡蛋”。
4. 系统展示该菜所需食材、调料和做法。
5. 用户点击“加入今日菜单”。
6. 用户继续添加其他菜，例如“青椒土豆丝”“紫菜蛋花汤”。
7. 用户进入“购物清单”页面。
8. 系统自动合并所有菜品需要购买的材料。
9. 用户按购物清单去买菜。
10. 回家后打开对应菜品详情，跟着步骤做饭。

### 示例

用户选择“西红柿炒鸡蛋”后，页面展示：

食材：

- 西红柿 2 个。
- 鸡蛋 3 个。
- 小葱 1 根，可选。

调料：

- 食用油 适量。
- 盐 2 克。
- 白糖 3 克，可选。

做法：

1. 西红柿洗净切块。
2. 鸡蛋打散，加少量盐搅匀。
3. 热锅加油，倒入鸡蛋炒至凝固后盛出。
4. 锅中补少量油，放入西红柿翻炒出汁。
5. 倒回鸡蛋，加盐和少量白糖调味。
6. 翻炒均匀后出锅。

## 7. 页面规划

### 7.1 首页

目标：让用户快速找到想吃的菜。

模块：

- 顶部搜索框。
- 菜品分类入口，例如家常菜、快手菜、汤、肉菜、素菜、早餐。
- 推荐菜品列表。
- 今日菜单入口。
- 购物清单入口。

### 7.2 菜品列表页

目标：按分类浏览菜品。

字段：

- 菜品图片。
- 菜名。
- 耗时。
- 难度。
- 标签，例如“新手友好”“10 分钟”“下饭菜”。
- 加入今日菜单按钮。

### 7.3 菜品详情页

目标：让用户明确知道该买什么、怎么做。

模块：

- 菜品图片和基础信息。
- 食材清单。
- 调料清单。
- 做法步骤。
- 小贴士。
- 加入今日菜单按钮。

### 7.4 今日菜单页

目标：管理本次准备做的菜。

功能：

- 展示已选择的菜。
- 支持删除菜品。
- 支持调整人数。
- 一键生成购物清单。

### 7.5 购物清单页

目标：帮助用户买菜。

功能：

- 按食材、调料分组。
- 相同材料自动合并。
- 可勾选已购买。
- 可显示“家中可能已有”的调料，例如盐、油、酱油。
- 后续可支持复制清单、分享给家人。

## 9. 技术路线建议

考虑到你是技术小白，建议第一版采用简单、资料多、容易部署的方案。

### 第一阶段：纯前端原型

技术建议：

- React 或 Next.js。
- Tailwind CSS 做样式。
- 菜谱数据先放在 JSON 文件中。
- 今日菜单和购物清单先存在浏览器 localStorage。

优点：

- 不需要先学复杂后端。
- 可以快速做出可点击、可使用的网站。
- 适合 vibe coding 快速迭代。

### 第二阶段：加入后端和数据库

技术建议：

- Next.js API Routes 或独立 Node.js 后端。
- 数据库使用 Supabase、PostgreSQL 或 Firebase。
- 支持用户登录和云端保存菜单。

可增加功能：

- 用户收藏菜品。
- 家庭成员共享菜单。
- 自定义菜谱。
- 上传菜品图片。

### 第三阶段：产品化上线

技术建议：

- 前端部署到 Vercel。
- 数据库使用 Supabase。
- 图片使用对象存储或 Supabase Storage。
- 域名绑定到 Vercel。

可增加功能：

- 一周菜单规划。
- 营养估算。
- 食材库存管理。
- 根据已有食材推荐菜品。
- AI 生成菜谱和替代食材建议。

## 10. 前端设计方向

整体风格建议：清爽、家庭感、实用，不做复杂炫技。

### 视觉关键词

- 干净。
- 温暖。
- 易读。
- 像家庭厨房助手，而不是外卖 App。

### 页面风格

- 白色或浅色背景。
- 用绿色、番茄红、米黄色做少量强调色。
- 菜品图片要真实、清晰。
- 按钮文案直接，例如“加入今日菜单”“生成购物清单”。
- 移动端优先，因为用户买菜和做饭时更可能用手机。

### 关键交互

- 搜索框始终容易找到。
- 菜品卡片支持一键加入菜单。
- 购物清单支持勾选。
- 做法步骤要分段显示，适合边做边看。

## 11. 后端与存储规划

### MVP 阶段

不需要后端。

数据存储方式：

- 菜谱数据：项目里的 JSON 文件。
- 用户选择的今日菜单：浏览器 localStorage。
- 勾选状态：浏览器 localStorage。

### 正式阶段

需要后端和数据库。

建议数据库表：

1. users 用户表。
2. recipes 菜品表。
3. ingredients 食材表。
4. recipe_ingredients 菜品食材关联表。
5. menus 用户菜单表。
6. menu_items 菜单明细表。
7. shopping_lists 购物清单表。
8. shopping_list_items 购物清单明细表。

### 数据库选择

对新手更友好的选择：

- Supabase：适合快速拥有数据库、登录、文件存储。
- Firebase：适合轻量应用，但结构化查询不如 SQL 直观。

建议优先 Supabase。

## 13. 开发里程碑

验收标准：

目标：用户能看到菜品所需材料和做法。

### 里程碑 3：今日菜单

目标：用户能选择多个菜。

- 增加“加入今日菜单”功能。

- 使用 localStorage 保存状态。

### 里程碑 4：购物清单

目标：用户能根据菜单买菜。

## 16. 风险与注意事项

### 数据准确性

菜谱的材料用量和步骤需要尽量准确，否则用户实际做饭会失败。初期建议先录入少量高质量菜谱，不要追求数量。

### 食材单位合并

购物清单合并会遇到单位问题，例如“鸡蛋 2 个”和“鸡蛋 3 个”容易合并，但“葱 1 根”和“葱 适量”不容易合并。MVP 可以先只做简单合并。

### 用户体验

做饭时用户可能一只手操作手机，所以按钮要大，步骤要清楚，页面不要太复杂。

### 版权问题

不要直接复制其他网站的大量菜谱内容和图片。可以自己整理菜谱文字，图片优先使用自有图片、免费版权图片，或后续用 AI 生成图片。

## 18. 当前推荐结论

第一版应专注于一个清晰闭环：

用户选菜 -> 查看材料和做法 -> 加入今日菜单 -> 生成购物清单 -> 买菜做饭。

不要一开始做完整平台，也不要急着接后端。先用纯前端和本地数据做出可用原型，验证流程是否顺畅。等原型好用之后，再逐步增加登录、数据库、家庭共享和 AI 推荐。
