# Humi Collaboration Identity and History Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让登录用户自动携带 Humi 身份、游客自动获得请求范围身份，并把“谁做了什么、结果如何”保存为可读、可合并的家庭协作历史。

**Architecture:** Store 使用统一 `collaborationEvents` 作为 append/update-safe 历史源；现有 crave/grocery/wish 实体继续承载业务状态，但每个提交同步写一条规范参与者快照。公开落地页只提交业务动作，不询问姓名；H5 已有完整 session 时传正式身份，游客由服务端在当前请求内分配别名。家庭协作记录页从受鉴权的历史 API 读取自然语言事件。

**Tech Stack:** Node.js、HumiStore JSON、React 19、Playwright、Node `assert`。

## Global Constraints

- 已登录参与者使用服务端 session 对应的用户 ID、昵称和头像；不得信任客户端伪造的正式身份字段。
- 游客只在首次提交动作时创建请求范围 `guestParticipantId`；仅打开链接不写参与者。
- 游客不填写姓名或家庭关系；展示名由服务端生成 `游客 1`、`游客 2`。
- 游客 ID 不得跨请求追踪；client storage key 必须包含协作类型与 token。
- 同一请求、同一参与者、同一动作重试必须幂等更新，不能制造重复历史。
- 游客主动登录合并后保留原始 `createdAt`、动作和 payload，并记录 `mergedAt`/`mergedFromGuestId`。
- 游客动作和协作 `/join` 身份绑定都不会自动成为家庭成员；只有用户明确接受家庭邀请并通过家庭邀请 endpoint 才建立成员关系。
- 历史输出不包含 token、ownerSecret、OpenID、手机号或内部错误码。

### Task 1: Canonical Collaboration Event Model

**Files:**
- Create: `scripts/check-collaboration-identity.mjs`
- Modify: `api/store.js`
- Modify: `package.json`

**Interfaces:**
- Produces:
  - `recordCollaborationEvent(input) -> CollaborationEvent`
  - `mergeGuestCollaborationEvents({ requestType, requestId, guestParticipantId, user }) -> CollaborationEvent[]`
  - `getHouseholdCollaborationEvents(userId, householdId, { limit }) -> CollaborationEvent[]`

Canonical event:

```js
{
  id,
  householdId,
  requestType: "crave" | "grocery" | "wish",
  requestId,
  participantType: "user" | "guest",
  participantId,
  displayNameSnapshot,
  avatarSnapshot,
  actionType,
  payload,
  createdAt,
  updatedAt,
  mergedAt: "",
  mergedFromGuestId: "",
}
```

- [x] **Step 1: Write failing event-model tests**

Assert guest aliases are sequential per request, the same guest/action retry preserves `id` and `createdAt`, a second request starts again at `游客 1`, and history access requires formal membership. Merge must produce `participantType: "user"`, current user snapshots, preserved action/payload/createdAt, and a non-empty `mergedFromGuestId`.

- [x] **Step 2: Verify RED**

Run: `npm run validate:collaboration-identity`

Expected: script missing or `recordCollaborationEvent` undefined.

- [x] **Step 3: Initialize and normalize persisted data**

Add `collaborationEvents: []` to default data and normalize missing arrays during `load()`. Event idempotency key is:

```js
`${requestType}:${requestId}:${participantType}:${participantId}:${actionType}`
```

Guest alias index counts distinct guest participant IDs already present for the same request, not global data.

- [x] **Step 4: Implement event recording, merge and history authorization**

`payload` is copied through a per-action sanitizer, not arbitrary nested user input. History returns newest first and clamps limit to `1..100`. Merge is idempotent: calling it twice yields the same event IDs and does not change original `createdAt`.

### Task 2: Record Crave, Grocery and Wish Actions

**Files:**
- Modify: `api/store.js`
- Modify: `api/server.js`
- Modify: `scripts/smoke-humi-api.mjs`

**Interfaces:**
- Consumes: Task 1 event functions.
- Produces public action bodies with `{ guestParticipantId }` or authenticated action bodies whose identity is derived from bearer session.

- [x] **Step 1: Add failing API smoke assertions**

For each collaboration type:

1. GET share creates no event.
2. Guest submit without a guest ID returns one generated by server and records `游客 1`.
3. Retry with returned guest ID updates the same event.
4. A signed-in submit ignores fake client `displayNameSnapshot` and records the session user.
5. Guest action alone does not add a household member.

- [x] **Step 2: Verify RED**

Run `npm run validate:api`; expected failure because responses lack canonical participant fields.

- [x] **Step 3: Extend store action signatures**

Each add method accepts a trusted participant argument separately from business input:

```js
addCraveVote(token, vote, participant)
addGroceryShareClaim(token, claim, participant)
addWishShareEntry(token, wish, participant)
```

For guests, participant is `{ type: "guest", id: sanitizedOrGeneratedId }`; for users it is created after `requireAuth` and `store.getUser`. Existing `memberName`, `temporary`, `memberId` fields remain response-compatible but are derived from canonical identity.

- [x] **Step 4: Allow optional auth on public action handlers**

Add `optionalAuth(request)` that returns a validated user or null. An invalid supplied bearer token returns 401 instead of silently becoming a guest. Server response includes `{ request, participant: { type, id, displayName, avatar } }`.
