# Humi 原生小程序骨架迁移设计

## 决策

Humi 从“微信小程序 WebView 壳承载整套 H5”迁移为“原生小程序承载核心产品，H5 只承载低频内容页”的混合架构。迁移不推翻自建 Humi API、菜谱数据、家庭模型、推荐规则或 MealRun 状态机，也不恢复 Supabase。

本设计同时收口以下已确认问题：

- 【今晚】推荐在固定两组之间循环；
- 菜单、清单和海报的微信分享均无法完成原生页面交接；
- 海报点击“换一种样式”后经常没有视觉变化；
- 晚间轻确认在今晚安排阶段过长、过早，并与“上桌了”重复；
- H5 启动、微信登录和图片首屏加载偏慢；
- WebView 到微信原生能力的桥接缺少可观测性和可靠回退。

微信流量主广告不属于本轮范围。后续必须另做产品位置、频控、资格、审核与收益设计，不在原生骨架中预埋广告组件或广告位 ID。

## 为什么需要换骨架

2026-07-22 生产日志显示，菜单分享接口连续八次返回 `201`，清单分享和海报上传也返回 `201`，但真机没有进入微信原生发送页。故障发生在 H5 完成业务请求之后、微信小程序原生页面接手之前。

当前登录、分享、订阅、保存图片、后台恢复和页面栈都跨越 WebView 桥接。桥接成功回调只能证明调用被接收，不能证明目标页面已经可见；客户端失败也不会进入 Nginx 或 API 日志。继续给每项能力增加局部补丁，会放大不可观测的最后一公里风险。

H5 仍适合承载菜谱阅读、长内容和低频分析页，但不再作为小程序的应用控制器。

## 目标架构

### 原生小程序负责

- App 启动、静默微信登录、身份完善和会话恢复；
- 五个一级入口：今晚、发现、计划、清单、我的家；
- 行动力选择、平衡推荐轮换、就做这顿、开始做、整顿饭时间线和上桌确认；
- 家庭切换、成员列表、邀请、协作任务、菜单/清单分享和微信提醒；
- 原生分享卡片、海报保存/发送、页面栈和前后台生命周期；
- 离线状态、待同步动作和错误恢复；
- 小程序启动性能、分包、资源缓存和真机诊断。

### H5 保留

- 完整菜谱文章和长步骤阅读页；
- 低频统计、营养回顾和历史兼容页；
- 迁移期未原生化的辅助页面；
- 非微信浏览器的网页版访问。

H5 页面只能通过一个通用原生 `web-content` 页面进入。它不再负责登录、一级导航、分享、订阅、家庭切换或 MealRun 状态控制。

### 自建 Humi API 负责

- 用户、身份、家庭、成员、家庭共享状态和权限；
- 晚餐推荐上下文、轮换状态、MealRun、协作任务和提醒；
- 菜单/清单只读分享快照；
- 跨设备状态版本、幂等和审计事件；
- 原生启动所需的聚合 bootstrap 数据。

## 原生应用信息架构

原生 `tabBar` 保持五个已建立心智的入口：

1. `今晚`：行动力、推荐、当前 MealRun、早餐/午餐轻入口；
2. `发现`：原生双列菜品流、分类、搜索，菜谱正文可进入 H5 内容页；
3. `计划`：多日餐次安排和汇总清单；
4. `清单`：采购项、家里已有线索、买菜协作；
5. `我的家`：家庭概览、成员、邀请、协作记录、设置。

二级原生页包含 `cooking`、`meal-task`、`share`、`poster`、`reminder`、`identity`、`household-settings`、`family-activity` 和 `web-content`。

N1 先为已注册的 cooking、settings、invite、task、recipe 和 web-content 分包路径提供合法四文件 `page-state` 占位，文案仅为“功能将在后续阶段启用”。后续能力任务在这些占位上实现或替换真实交互；N1 不调用 API，也不提前启用 N2/N3 业务。

## 启动与登录

1. `App.onLaunch` 读取原生安全存储中的 Humi session；
2. 有效 session 立即并行请求 `/bootstrap` 和必要缩略图，不等待 H5；
3. 无 session 或 session 失效时调用 `wx.login`，以 code 请求 `/auth/wechat/login`；
4. 登录成功但身份未完善时进入原生身份页；
5. bootstrap 失败时展示原生离线缓存和明确重试，不跳入空白 WebView；
6. `401` 只允许一次静默重新登录和原请求重放；非幂等请求必须带 idempotency key 才能重放；
7. 登录、bootstrap 和首屏图片分别计时，日志不记录昵称、token 或自由文本。

原生存储只保存 session、版本化缓存和离线队列。家庭共享状态的事实源始终是 Humi API。

历史公开分享链接是启动路由的受控例外：`crave`、`grocery`/`groceryShare`、`menuShare`、`wishShare`、`invite` 与 `mealTask` 携带 token 时，boot 或历史 `pages/index/index` shim 在 core-shell flag 与 bootstrap 之前，直接 reLaunch 既有原生 `share` landing，并保留 token 与 `shareSource`。该页面不进入五个核心 tab，也不读取或修改家庭共享状态；没有已知 token 的普通启动和深链仍按 bootstrap flag 进入原生 core shell 或 H5 compatibility shell。

## 平衡推荐轮换

沿用已确认的 C 方案和 `2026-07-22-humi-balanced-recommendation-rotation-design.md`：

- 忌口、过敏、认证和行动力是不可放宽的硬约束；
- 在高分候选窗口中轮换，而不是全库随机；
- scope 为家庭、日期、行动力档位和推荐上下文指纹；
- 连续五组不重复任何一道菜；候选耗尽后避开最近两组再开启新周期；
- `下次还想吃`、`可以换换`、`太费劲`影响未来权重，但不破坏当前轮换；
- 原生端和 H5 端通过同一推荐服务合同获得结果，不各自维护互相冲突的排名。

游客可用本地打包的 30 道认证菜获得三档方案；登录家庭由 API 返回家庭化结果和服务端轮换游标。原生端保存最近展示 ID，用于离线和防止重复点击。

## 做饭和轻确认

“就做这顿”只创建 `planned` MealRun，“开始做”进入 `cooking`，“上桌了”才进入 `completed`。

- 从【今晚】和【今晚安排】移除完整 `DinnerLogPanel`；
- 新原生流程在时间线末尾只保留一个明确的“上桌了”；
- 迁移期旧 H5 流程在用户点击“开始做饭”之后才展示紧凑确认条；
- 未确认上桌不计成功，也不计失败；
- 临时改计划可显式放弃，但不破坏连续记录；
- 上桌后再收集一次轻反馈，避免决策页和烹饪页重复询问。

所有正式家庭成员可开始、推进和上桌确认；只有所有者能修改家庭菜单和设置。

## 分享与海报

### 生产止血

原生迁移完成前，H5 热修必须：

- 菜单、清单或海报 token 只生成一次，原生交接失败时复用；
- 不把 JS SDK 的 `success` 回调单独视为目标页已打开；
- 未观察到页面离开时继续走可靠的 `redirectTo/reLaunch`；
- 记录匿名 bridge 阶段、方法、结果、耗时和错误码；
- 分享请求 `401` 时静默登录并最多重试一次；
- 不重复创建分享快照或海报上传。

### 原生完成态

原生一级页直接创建分享快照并使用 `<button open-type="share">` 与 `onShareAppMessage`，不再经 H5 跳转原生分享页。正式家庭成员可以分享现有菜单或清单的只读快照；该权限不包含修改菜单、家庭设置或邀请成员。

海报由原生页面持有明确 `styleId`。点击换风格按照固定序列切换到下一个不同模板，不使用可能重复上一款的随机抽样。只有一个模板的海报不显示“换一种样式”。

## 状态、离线和并发

- `/bootstrap` 返回 `stateVersion`、用户、家庭列表、当前家庭、家庭共享状态、当前晚餐执行和能力开关；
- 所有状态修改携带当前 `stateVersion`，冲突返回 `409` 和服务器最新版本；
- MealRun 使用现有幂等状态接口；
- 原生离线队列只接受白名单动作，按家庭和 MealRun 排序重放；
- 菜单修改、家庭设置和所有权操作不做离线乐观覆盖；
- 分享创建、完成、反馈等重复点击使用 idempotency key；
- 切换家庭会清空页面级派生状态，但保留按家庭隔离的缓存。

## 性能

- 启动不下载 React 主包；首屏先渲染原生骨架和本地缓存；
- 原生主包只包含启动、今晚和公共组件，其余一级页和内容能力分包；
- 发现页只加载缩略图，菜谱正文按需加载；
- 同一图片 URL 复用微信缓存，失败显示本地灰阶占位，不使用空白圆；
- bootstrap、推荐、缩略图和 MealRun 恢复分别设置性能预算和埋点；
- H5 继续拆分 922 kB 级主 chunk，避免网页版和 `web-content` 页面长期背负整包。

## 灰度与回滚

使用两层开关：

- 小程序本地 `nativeShellCandidate` 决定候选版本是否具备原生页；
- 服务端 `nativeShellEnabled` 与家庭白名单决定账号是否进入原生核心流程。

关闭服务端开关保证所有无 token 的 core-shell 启动和未知深链回到 `pages/legacy/index`；已知公开 share token 保持为既有原生 `share` landing 的 token-preserving 例外，因此不需要游客先登录读取 bootstrap。

灰度顺序为内部账号、5 个家庭、20 个家庭、20%、50%、100%。任何阶段出现登录成功率、分享到达率、开始做饭率或 MealRun 完成率明显下降，立即把服务端开关切回 H5 壳，不回滚数据 schema。

每个阶段独立保留旧 H5 入口至少一个稳定版本。正式发布、微信审核、生产 API 部署和白名单扩大均需单独 checkpoint。

## 可观测性

白名单事件包括：

- `native_boot_started/completed/failed`；
- `native_login_started/completed/failed`；
- `bootstrap_completed/failed`；
- `share_snapshot_created`、`native_share_page_visible`、`native_share_cancelled/failed`；
- `poster_style_changed`、`poster_saved/shared/failed`；
- 已有 MealRun 开始、完成和放弃服务端事件。

事件只包含匿名 session、家庭 ID、页面、阶段、耗时、错误码、版本和必要业务 ID，不上传昵称、token、菜谱备注或家人自由文本。原始事件保留 180 天。

## 验收

- 冷启动、热启动、session 有效、session 过期、首次身份完善和离线启动均有真机证据；
- 登录完成后不先加载整套 H5，首屏可交互不依赖 WebView；
- 三档推荐分别连续换五组不重复，所有硬约束零违规；
- 菜单、清单、邀请、任务、海报五类分享在真实联系人中完成发送和接收；
- 海报连续换两次必定显示不同模板；
- 晚间安排无冗长轻确认，只有开始后出现一次上桌确认；
- 后台恢复、绝对计时器、离线进度和网络恢复同步通过；
- 五个一级入口、家庭权限、多家庭隔离和 H5 内容页回退通过；
- 390×844 iOS 微信真机、Android 微信真机和微信开发者工具均通过；
- 功能开关关闭时可立即回到现有 H5 稳定流程。

## 范围外

- 微信流量主广告、插屏广告、激励视频和广告位申请；
- iOS/Android 独立 App；
- 早餐、午餐完整烹饪执行模式；
- 138 道菜全部 cookAssist 认证；
- 支付、订单、会员权益和交易平台；
- 一次性删除 H5 或取消网页版。
