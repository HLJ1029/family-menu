# Humi 微信身份、家庭与协作体验重构设计

日期：2026-07-19
状态：已确认；Phase 1–4 本地候选与本地门禁已完成；微信真机、生产迁移/部署和外部 provider 删除仍是独立授权门禁
风险等级：High（登录、身份、生产数据、核心前后端联调、小程序关键路径）

实施进度（2026-07-20）：Phase 1“身份与启动链路”、Phase 2“家庭生命周期与家庭客厅”、Phase 3“协作身份与历史”均完成本地候选及独立门禁；Phase 4 已完成本地迁移 audit/dry-run/apply-to-copy 工具、回滚 runbook、仓库 Supabase runtime/依赖/配置清退、真机证据检查器与最终本地矩阵，审计见 `docs/humi-identity-family-collaboration-final-audit.md`。本轮没有真机微信操作、生产部署或写入、生产 backup/dry-run/apply/rollback、provider 操作或外部 Supabase 数据/Secrets 删除；本地 GO 不等于生产 rollout 完成。

## 1. 目标

本设计把 Humi 的微信登录、用户身份、家庭生命周期和协作记录重构为一条完整链路，解决以下生产体验问题：

- 首次打开小程序即被静默登录，用户没有主动选择。
- `wx.login` 仅获得 OpenID，却被产品当成完整用户资料，导致所有人显示相同头像和“微信用户”。
- 打开小程序即产生账号壳，账号数不能代表真实注册或使用人数。
- 系统自动创建“我的家”，但用户不知道家庭能做什么，也没有明确的建家动作。
- “我的家”混入画像、算法、验证、云同步和协作等大量信息，缺少可理解的主任务。
- 协作只强调动作数量，弱化了“谁做了什么、结果如何”的家庭感知。
- 产品仍保留 Supabase 运行链路，与“自建 Humi API 为唯一正式后端”的目标不一致。

重构后的用户心智应当是：

```text
我主动进入 Humi
→ 我选择登录或游客体验
→ 登录后建立自己的身份
→ 我主动创建或加入一个家
→ 家庭成员围绕菜单、想吃和买菜协作
→ “我的家”清楚展示家人、进行中的协作和家庭设置
```

## 2. 当前生产基线

2026-07-19 只读核查结果：

- 自建 Humi API 中有 18 个微信账号和 18 个唯一微信身份。
- 没有 mock 身份、重复 OpenID 映射或孤儿身份记录。
- 7 个账号形成家庭空间，6 个家庭写入过云端状态。
- 0 个账号绑定手机号，独立 profile 记录为 0。
- 当前普通启动会先执行 `wx.login`，后端按 OpenID 查找或创建用户，再把 Humi 会话交给 H5。
- H5 收到会话后直接标记登录和引导完成，因此首次用户看不到登录页或身份完善页。
- GitHub Pages 构建仍注入 Supabase URL 和 Anon Key，H5 启动仍尝试恢复 Supabase Auth 会话。

该基线说明问题来自身份生命周期设计，而不是多个用户共享同一个登录态。

## 3. 设计原则

### 3.1 用户主动建立正式身份

- 普通启动不得自动调用微信登录或创建后端用户。
- 只有用户主动点击“微信登录”后才调用 `wx.login`。
- 游客可以完成核心菜单体验，并且不产生正式账号记录。
- 已登录且会话有效的老用户可自动恢复会话，不重复展示登录页。

### 3.2 身份建立只做一次，协作过程零打断

- 首次微信登录后完成一次昵称与头像设置。
- 日常协作不再重复询问姓名或家庭关系。
- 登录用户自动携带正式身份；游客自动携带本次协作范围内的游客身份。

### 3.3 家庭必须由用户明确创建或加入

- 登录不等于建家。
- 读取用户状态不得产生家庭副作用。
- 家庭只能由明确的“创建家庭”或“加入家庭”动作产生。

### 3.4 家庭页面只服务家庭任务

- “我的家”首先回答：这是哪个家、有哪些人、正在一起做什么。
- 算法状态、测试数据、技术术语和冗长画像不得出现在家庭首页。
- 管理能力进入明确的子页面，不在一个长页面中堆叠。

### 3.5 自建后端是唯一正式事实来源

- Humi API 负责身份、资料、家庭、协作和状态同步。
- 正式构建不再注入或初始化 Supabase。
- Supabase 旧数据在清退前只读备份，不作为新功能依赖。

## 4. 首次进入与登录状态机

### 4.1 普通首次打开

首次打开显示 Humi 登录落地页：

- 主操作：`微信登录`。
- 次操作：`先体验 Humi`。
- 不显示“已登录”，不调用 `wx.login`，不创建用户或家庭。

### 4.2 游客进入

用户选择游客体验后：

- 进入今晚菜单主流程。
- 本机保存游客菜单和清单。
- 顶部或“我的家”显示明确的游客状态，不伪装为登录用户。
- 需要发起家庭协作、保存到云端或创建家庭时，再说明登录价值并允许用户主动登录。

### 4.3 用户主动微信登录

用户点击“微信登录”后：

1. 原生小程序调用 `wx.login` 获取临时 code。
2. Humi API 使用 code 换取微信身份并创建或恢复 Humi 用户。
3. 原生层取得短期、一次性的 H5 会话交换票据。
4. H5 使用票据换取正式 Humi 会话，立即从地址栏移除票据。
5. 长期 access token 不直接放入 WebView URL。
6. 新用户进入一次性的身份完善页；资料完整的老用户继续进入家庭判断。

### 4.4 会话恢复与失效

- 有效会话：直接恢复用户身份，不重复出现登录落地页。
- 会话失效：保留本机游客数据，显示“登录已过期”，由用户主动重新验证。
- 登录接口失败：停留在登录页，提供重试和游客体验，不显示虚假的成功状态。
- 用户退出：清除正式会话，但不删除本机游客菜单；返回游客模式。

## 5. Humi 用户身份完善

微信登录只建立 OpenID 对应关系，不代表资料完整。首次成功登录后显示一次“你的 Humi 身份”页面。

### 5.1 字段与交互

- 昵称：必填，使用微信小程序 `input type="nickname"` 能力提供微信昵称建议。
- 头像：可使用 `button open-type="chooseAvatar"` 选择微信头像，也可保留系统分配的 Humi 默认头像。默认头像从已批准的 Humi 头像集合中按用户 ID 稳定分配，避免所有账号显示同一头像。
- 确认按钮：`进入 Humi`。
- 手机号：不在首次身份完善中强制；保留在账号设置中，由用户主动绑定。

### 5.2 资料状态

用户资料具有明确状态：

- `incomplete`：只有登录身份，尚未确认昵称。
- `complete`：昵称已确认，头像使用用户选择或系统默认值。

H5 不再通过“是否存在 session”推断资料和引导均已完成。

## 6. 家庭生命周期

### 6.1 登录但没有家庭

身份完善后进入“创建或加入家庭”页面：

- 主操作：`创建我的家`。
- 次操作：`通过邀请加入`。
- 页面用简短语言说明家庭价值：共享菜单、一起决定想吃什么、协作买菜。
- 不自动创建名为“我的家”的家庭。

### 6.2 创建家庭

创建过程为一个明确页面：

- 家庭名称必填，默认建议“我们家”，用户可修改。
- 前端提交、Store 和 HTTP API 都必须拒绝 trim 后为空的家庭名称；HTTP 返回 `400 household_name_required`。
- 确认后创建家庭，当前用户成为主厨。
- 创建成功进入“家庭客厅”，并突出“邀请家人”下一步。
- 创建失败停留原页并保留输入，不在前端伪造家庭成功状态。

### 6.3 加入家庭

- 邀请卡片打开时可先查看家庭名称和邀请人。
- 游客可以浏览允许公开的邀请落地信息，但正式加入需要主动登录。
- 登录并确认加入后成为正式成员，进入该家庭的“家庭客厅”。
- 邀请失效、撤销或已使用时显示明确状态，不退回空白页。

### 6.4 家庭角色

- 主厨：邀请成员、管理成员、修改家庭设置、创建或切换家庭。
- 家庭成员：查看家庭、参与菜单和买菜协作、维护自己可编辑的信息。
- 游客参与者：仅在具体协作中留下匿名动作，不自动成为家庭成员。

## 7. “我的家”信息架构

已选择 A 方案“家庭客厅”。“我的家”是家庭 Hub，不再是账号与数据仪表盘。

### 7.1 家庭首页

首页仅保留四个区块：

1. 当前家庭
   - 家庭名称。
   - 当前用户角色。
   - 成员头像与人数。
2. 家庭操作
   - 成员管理。
   - 家庭设置。
   - 协作记录。
   - 账号设置。
3. 正在一起做
   - 正在进行的想吃征集。
   - 正在进行的买菜清单。
   - 其他仍需用户行动的家庭协作。
4. 家庭偏好
   - 一句话概括人数、主要口味与忌口。
   - 点击进入家庭偏好编辑。

邀请家人是“当前家庭”卡片上的明确主操作，不与四个家庭子页面混成一个不可理解的信息面板。当前家庭切换、退出后重新建家或其他 `family.id` 变化时，家庭内部导航必须在同一次渲染中回到客厅，不得短暂显示上一家庭的设置页。

每个协作卡片必须展示清楚的任务、参与进度和下一动作，例如“今晚想吃什么 · 1 人已回复 · 去看看”。

### 7.2 独立子页面

#### 成员管理

- 成员头像、昵称、角色和加入状态。
- 主厨可邀请或移除成员。
- 普通成员不能看到无权限操作。

#### 家庭设置

- 家庭名称。
- 家庭共同偏好。
- 切换家庭。
- 退出家庭；主厨退出前必须满足所有权转移或解散规则。

#### 协作记录

- 已结束和进行中的征集、想吃与买菜任务。
- 参与者、动作、结果和完成时间。
- 默认按时间倒序，不展示技术 token 或内部状态名。
- 活动文案和 DOM 不得出现 owner secret、participant key、household ID、token 值或这些内部字段名。

#### 账号设置

- 用户头像与昵称。
- 手机号绑定状态。
- 退出登录。
- 隐私政策和账号相关入口。

### 7.3 从首页移除

以下内容不得出现在“我的家”首页：

- AI 模型、推荐层级、试用额度等算法状态。
- 验证事件、开发测试记录和内部审计信息。
- “云同步已开启”等技术状态；只用“已保存到家”表达用户结果。
- 冗长的饮食画像与来源统计。
- 与当前家庭动作无关的历史信息流。

## 8. 协作身份与记录

### 8.1 身份选择规则

协作动作前不显示身份表单：

- 已登录：使用 Humi 用户 ID、当前昵称、头像和家庭成员角色。
- 游客：首次提交协作动作时才为当前协作生成不透明的 `guestParticipantId`，显示为“游客 1”“游客 2”等请求内别名；仅打开链接不产生参与者记录。
- 游客标识只在当前协作请求中有效，不用于跨协作追踪。

### 8.2 协作记录结构

每个协作动作保存：

- `participantType`：`user` 或 `guest`。
- `participantId`：用户 ID 或请求范围的游客 ID。
- `displayNameSnapshot`：动作发生时的展示名。
- `avatarSnapshot`：动作发生时的头像引用；游客使用统一游客头像。
- `actionType`：投票、想吃、认领、完成、拒绝等。
- `payload`：该动作必要的业务内容。
- `createdAt`：服务端时间。

展示名和头像快照用于保持历史可读性；正式用户 ID 用于关联当前成员资料。

### 8.3 游客登录合并

游客在当前协作流程中主动登录后：

- 客户端提交当前请求的游客 ID。
- 服务端校验游客 ID 属于当前协作并尚未被其他用户认领。
- 将本次协作记录关联到正式用户，同时保留原始时间和动作。
- 合并失败不丢失游客记录，也不阻断用户继续使用。

### 8.4 家庭感知

家庭首页与协作详情使用自然语言：

- `小禾想吃番茄炒蛋`。
- `游客 1 已买到鸡蛋`。
- `2 人已回复，还有 1 位家人未参与`。

不再只显示“收到 2 条记录”或内部状态值。

## 9. 后端边界

### 9.1 唯一事实来源

自建 Humi API 是正式环境的唯一身份与家庭后端。完成重构后：

- 正式构建不注入 Supabase 环境变量。
- 前端不初始化 Supabase client 或恢复 Supabase Auth。
- 邮箱登录、Supabase 家庭同步和 Supabase 埋点退出正式运行路径。
- Supabase 依赖、旧环境变量和失效文档在备份验证后移除。

### 9.2 无副作用读取

以下读取操作不得创建用户、家庭或成员：

- 读取当前用户资料。
- 读取家庭列表。
- 读取家庭状态。
- 打开游客协作链接。

只有明确的登录、创建家庭、接受邀请和游客转正式用户动作可以写入对应实体。

### 9.3 概念接口

实现计划应维持下列明确边界：

- 身份：登录、会话交换、刷新、退出。
- 用户资料：读取与更新昵称、头像、资料完成状态。
- 家庭：创建、读取、更新、切换、成员管理、邀请加入。
- 协作：创建请求、游客参与、登录用户参与、游客记录合并、历史查询。
- 家庭状态：菜单、清单、偏好等现有同步能力，读取时不自动建家。

本设计不决定生产存储引擎更换；JSON 文件迁移到数据库属于独立基础设施决策，不与本轮用户体验重构捆绑。

## 10. 现有生产数据迁移

迁移遵循“不删数据、渐进补全身份、停止继续制造账号壳”。

### 10.1 迁移前保护

- 备份自建 Humi API 当前数据文件并记录校验值。
- 只读导出 Supabase Auth、profiles、families 和仍有数据的业务表计数。
- 不把 Supabase 数据自动覆盖到 Humi API。
- 迁移脚本必须支持 dry-run、幂等执行和回滚。

### 10.2 现有 Humi 用户

- 保留 18 个现有微信用户和身份映射。
- 昵称仍为“微信用户”或没有有效头像的账号标记为资料不完整。
- 下次有效登录时进入一次身份完善页。
- 不删除没有家庭的账号壳；长期清理需要单独的活跃度规则和用户确认。

### 10.3 现有家庭

- 保留 7 个现有家庭及全部家庭状态。
- 已有家庭用户完成身份后进入新的家庭客厅。
- 没有家庭的用户进入“创建或加入家庭”。
- 不通过读取状态补建缺失家庭；异常数据进入迁移报告人工核对。

## 11. 错误与边界状态

- 微信登录取消或失败：保留登录页和游客入口。
- 身份资料保存失败：保留用户输入并允许重试。
- 头像安全检测不通过：提示重新选择，昵称和其他输入不丢失。
- 建家失败：不显示“已建立我的家”。
- 邀请失效：显示邀请状态和返回 Humi 的入口。
- 协作游客提交失败：明确提示重试，不重复创建多条动作。
- 会话过期：本机数据保留，正式写入暂停，重新登录后恢复。
- 家庭被移除或权限变化：刷新家庭列表并给出自然语言提示。
- Supabase 清退验证失败：停止清退，不影响现有生产 Humi API 数据。

## 12. 验收标准

### 12.1 首次进入

- 全新安装首次打开不调用登录接口，不创建用户，不创建家庭。
- 用户看到微信登录和游客体验两个明确入口。
- 游客可以完成推荐、确认菜单和生成清单。

### 12.2 登录与身份

- 只有点击微信登录才产生微信登录请求。
- 新登录用户必须确认昵称，头像可选择微信头像或 Humi 默认头像。
- 完成资料后，头像昵称在顶部、我的家和协作记录中一致显示。
- 老用户会话有效时不重复登录；资料不完整时只补一次资料。

### 12.3 家庭

- 登录本身不创建家庭。
- 无家庭用户只看到“创建或加入家庭”。
- 创建成功后家庭客厅明确展示家庭名称、角色、成员和邀请入口。
- 建家草稿默认“我们家”；空白名称在前端、Store 和 API 三层均被拒绝，失败后保留原输入。
- 成员管理、家庭设置、协作记录和账号设置均为可导航的独立页面。
- 当前家庭身份变化时内部子页面同步回到家庭客厅，不出现旧家庭设置页的瞬时绘制。
- 家庭首页不出现技术术语、算法状态或测试信息。
- 改名、移除成员和转让主厨只更新家庭元数据，不丢失三餐记录、Crave、买菜或最近想吃协作状态。

### 12.4 协作

- 登录用户参与时自动显示真实 Humi 昵称和头像。
- 游客无需填写身份即可参与，并在当前协作中获得可区分的游客别名。
- 同一游客动作重试不会制造重复参与记录。
- 游客在当前流程登录后可安全合并记录。
- 主厨能看懂谁参与、做了什么以及最终结果。
- 协作记录不渲染 token、owner secret、participant key、household ID 或内部字段名。
- 通用协作认领只绑定参与身份，不创建正式家庭成员；正式加入家庭仅通过家庭邀请确认完成。

### 12.5 后端与迁移

- 正式 H5 构建不含 Supabase URL、Anon Key 或 Supabase Auth 初始化路径。
- 迁移前后用户、身份、家庭和家庭状态数量核对一致，新增明确迁移记录除外。
- 当前 6 个家庭云端状态可在新版本中读取。
- 生产数据备份、回滚步骤和 secret scan 均通过。

## 13. 实施分期

由于本设计同时涉及身份、家庭与协作，实施必须拆成可独立验收的阶段：

1. 身份与启动链路
   - 取消启动静默登录。
   - 建立主动登录、会话交换、资料完成状态和游客模式。
   - 清退正式运行时 Supabase Auth。
2. 家庭生命周期与家庭客厅
   - 去除读取时自动建家。
   - 建立创建或加入家庭流程。
   - 实现家庭客厅及四个独立子页面。
3. 协作身份与历史
   - 统一登录用户与请求范围游客身份。
   - 增加可读协作记录和游客登录合并。
4. 数据迁移与真机验收
   - 执行备份、dry-run 和现有账号资料补全标记。
   - 覆盖全新用户、现有用户、游客、主厨、成员和分享进入场景。

每个阶段都需要独立失败用例、API smoke、构建验证和微信真机验收；前一阶段未通过不得推进生产部署。

## 14. 非目标

本轮不包含：

- 支付、会员或额度产品重构。
- 更换自建后端生产存储引擎。
- 为游客建立跨协作长期画像。
- 强制手机号绑定。
- 重做菜单推荐、菜谱库或买菜清单业务逻辑。
- 批量删除现有账号或家庭数据。
